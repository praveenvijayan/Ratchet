
## Correctness bugs in the mechanism (fix first)

|#|Gap|Why it matters|Suggested fix|
|---|---|---|---|
|1|**Fresh claims get swept as stale on quiet repos.** [sweep-stale-claims.yml:38-45](https://claude.ai/epitaxy/.github/workflows/sweep-stale-claims.yml:38) uses the branch tip's commit date — but a fresh claim ref points at `main` HEAD, and agents don't push until gates pass. If `main`'s last commit is >2h old, every new claim looks stale within one sweep tick: the ref is deleted and the issue re-queued while the first agent is still building locally.|Double-claims, colliding pushes, duplicate PRs — it breaks the very atomicity the claim design is built on.|Measure staleness from the claim event (the `state:in-progress` labeled-event timestamp via the issue timeline), not the branch tip. Only fall back to commit dates when `ahead_by > 0`.|
|2|**`unblock-dependents` can promote criteria-less issues to `state:ready`.** plan-sync collapses state to one label, so "blocked AND no criteria" is stored as just `state:blocked` ([plan-sync.mjs:146](https://claude.ai/epitaxy/scripts/plan-sync.mjs:146)). When the blocker closes, [unblock-dependents.yml:53-55](https://claude.ai/epitaxy/.github/workflows/unblock-dependents.yml:53) blindly sets `state:ready`.|An agent picks an issue with no acceptance criteria — violating the system's own "criteria are the test plan" invariant.|Before promoting, re-check the issue body for a `- [ ]` criteria block; demote to `state:draft` if absent.|
|3|**No concurrency guard on plan-sync.** [plan-sync.yml](https://claude.ai/epitaxy/.github/workflows/plan-sync.yml) has no `concurrency:` group. Two merges touching `plan/**` in quick succession run two syncs in parallel; both see a slug as new and both create the issue.|Duplicate issues with the same `plan-id` marker; downstream blocker resolution becomes ambiguous.|Add `concurrency: { group: plan-sync, cancel-in-progress: false }`.|
|4|**No `blocked_by` cycle detection.** Two plan files blocking each other are both labeled `state:blocked` forever; `unblock-dependents` never fires and nothing warns.|Silent deadlock — the exact "silently stuck" failure the philosophy promises can't happen.|Detect cycles in plan-sync and fail the sync loudly; have `/ratchet-status` report cycles too.|
|5|**The lease isn't renewable.** Legitimate work >2h with no pushes (which the protocol _mandates_ — "only push after gates pass") is indistinguishable from a crash.|Long builds/large issues get swept mid-flight even after fixing #1.|Add a heartbeat (agent comments on the issue or updates the claim ref) or let agents push WIP to the claim branch since PR-open, not push, is the real gate.|

## Enforcement gaps — convention where a mechanism is needed

|#|Gap|Why it matters|Suggested fix|
|---|---|---|---|
|6|**Gates only run locally; PRs carry a self-reported checklist.** There is no CI workflow that runs `GATES.md` on the PR. The human reviewer trusts the agent's claim that gates passed.|Agents can (and do) hallucinate or shortcut verification. For a system whose bottleneck is human review, the reviewer's trust anchor must be machine-verified, not agent-asserted. This is the single biggest gap.|Add a `pr-gates.yml` workflow that runs the gates on every `agent/issue-*` PR, plus branch protection requiring it. Make `GATES.md` machine-readable (or generate a script from it) so local and CI gates are the same definition.|
|7|**Nothing actually protects `main`.** "Never merge, never push to main" is Hard Rule 6, but `/ratchet-init` doesn't configure branch protection. An agent with push access can violate every rule mechanically.|The human gate — the system's central premise — is enforced only by prompt obedience.|`/ratchet-init` should set branch protection: require PR, require the gates check, block force-push, optionally require review.|
|8|**Scope limits (~400 lines / ~6 files) are honor-system.** No mechanical check at PR time.|Oversized agent PRs are the #1 killer of review quality, and review is your bottleneck.|A tiny PR-size check in the gates workflow that fails (or labels) oversized PRs.|
|9|**Only `state:in-progress` is swept.** An issue stuck in `state:in-review` with a closed/abandoned PR, or `state:changes-requested` with a vanished agent, stays there forever.|Forward-only guarantee has holes in half the state machine.|Extend the sweep: `in-review` with no open PR → `ready`; `changes-requested` idle >N days → `ready` with a comment.|
|10|**No plan-file schema validation.** Priority isn't validated against `high|medium|low`(your own test uses`P1`/`P2` and it passes), so typos create nonstandard labels that sort as lowest priority silently.|

## Missing SDLC stages — "ship" ends at merge

|#|Gap|Why it matters|Suggested fix|
|---|---|---|---|
|11|**No release/deploy phase.** No versioning, changelog, release tagging, deploy gate, or environment story after merge.|"Build and ship high-quality product" — Ratchet currently builds; it doesn't ship.|Add an opt-in release lane: conventional-commit-driven changelog, tag/release workflow, and a deploy gate as a first-class post-merge step in the loop diagram.|
|12|**No rollback/incident path.** Forward-only has no answer for "the merged PR broke prod."|Every production system needs a defined revert path; today an agent would refuse to hotfix (Hard Rule 0) until a planning PR round-trips.|Define a hotfix protocol: a `priority:critical` fast lane and/or a sanctioned `revert` flow that still ends in a PR but skips the planning-PR round trip.|
|13|**No security gates.** `GATES.md` has format/typecheck/lint/test/build — no dependency audit, secret scan, or SAST. Agent-generated code arguably needs these _more_.|Whole vulnerability classes ship unchecked; a leaked secret in an agent commit is caught by no one.|`/ratchet-init` should detect and add `npm audit`/`pip-audit`, gitleaks/secret-scanning, and optionally a SAST step as gates 6–8.|
|14|**Prompt-injection surface in the unattended runner.** [ratchet-run.yml](https://claude.ai/epitaxy/.github/workflows/ratchet-run.yml) feeds issue bodies (compiled from plan files, but editable by anyone with issue access) to an agent holding a write-scoped PAT.|A malicious or careless issue edit becomes instructions to an agent with contents+PR write.|Restrict the runner to issues carrying the `plan-id` marker whose content matches the merged plan file; document the threat model; keep PAT scopes minimal.|

## Quality model & operability

|#|Gap|Why it matters|Suggested fix|
|---|---|---|---|
|15|**"Exactly one test per criterion" caps quality at the planner's imagination.** Edge cases, property/regression tests, and integration/e2e coverage are explicitly discouraged; non-functional requirements (performance, a11y, load, migrations) have no home in the plan format.|The bound is a great _stopping condition_ but a weak _quality floor_ — production defects live precisely in the cases the criteria didn't enumerate.|Keep the bound for unit tests but add optional plan sections (`## Non-functional`, `## Test notes`) and allow a coverage-threshold or mutation-testing gate as a backstop.|
|16|**The framework barely tests itself.** One test file covers plan-sync ordering; the sweep and unblock workflows — where bugs #1 and #2 live — are untested inline scripts.|The automation that guards everyone else's quality has the least of its own.|Extract workflow logic into `scripts/*.mjs` invoked by thin YAML, and test them the same way plan-sync is tested.|
|17|**No metrics.** No cycle time, gate-failure rate, rework rate, sweep frequency, or queue depth over time.|You can't tune a delivery mechanism you can't measure — e.g., you'd have caught bug #1 from sweep-comment frequency.|A read-only `ratchet-metrics` script/skill aggregating from issue timelines and PRs (all data already lives in GitHub — fits the philosophy).|
|18|**Plan files accumulate forever.** Closed issues' files stay in `plan/` and are re-scanned every sync; nothing archives them.|Noise grows linearly with project age; slug collisions and confusion follow in year two of a "multi-year" project.|Have `/ratchet-memory` (or a new sweep) move plans for closed issues to `plan/done/` — sync already resolves removed files via issue markers, so this is safe.|
|19|**Small hygiene items.** README layout block has duplicated lines ([README.md:50-67](https://claude.ai/epitaxy/local_ce46944f-d044-4aab-9e9e-4763151e5ec6)); `ratchet-run` hard-codes a stale model id (`claude-sonnet-4-6`); actions pinned to tags not SHAs; no PR template guiding the human review.|Individually minor; collectively they read as drift in a framework that sells discipline.|Quick cleanup pass; pin actions by SHA; add a PR template with a gates-results section the CI check can validate.|
